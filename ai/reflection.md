# Reflection

## 1. Which AI tools or models did you use?
I mainly used Claude (Opus 4.8). In the early stage, when I made the plan and drew the
UML diagram, I also looked at Kimi for some ideas.

## 2. Which prompt was the most useful? Why?
The most useful prompt was when I asked the AI to read all of our previous conversation
first and then answer. After it remembered the whole context of my project, its answers
fit my own code much better, and I did not need to explain everything again. This saved
me a lot of time.

## 3. Which AI-generated suggestion was wrong, incomplete, or misleading?
The AI made the most mistakes at the beginning, when it helped me write the plan. It
could not include every requirement at one time, so I had to check the requirement
document point by point and add the missing parts myself. The second common problem was
that the AI often put the add/delete/edit (CRUD) logic in the wrong class. For example,
it wanted to put data management inside the Admin class, but I decided it should go into
GameDataManager.

## 4. How did you check whether AI-generated code was correct?
My method was to compile after writing only a small part each time, and then print and
test it in the main console. If something was wrong, I could find it immediately instead
of waiting until the end.

## 5. What bugs did you fix yourself instead of asking AI to fix?
I fixed some bugs by myself, such as variable names that were repeated (name clash), and
some fields that the AI wrote twice by mistake. I found these from the compiler errors
and removed them myself.

## 6. What Java concept did you understand better after using AI?
I understood interfaces and abstract classes better after this project. Before, I only
knew they existed but did not really know why we need them. After using AI to design
the project, I saw that Searchable is an interface: it only says "you must have a
matchesQuery method" but does not write the code inside. Then Player, Team, Hero and
Equipment all write their own version. Because of this, SearchService can use one loop
to search any of them - it does not need to care which type it is. I also saw that
Person is an abstract class: it has real fields and a real constructor, but getInfo()
is abstract so every subclass must write its own version. The difference I now
understand is: an interface is a pure contract with no data, an abstract class can
have real fields and some real methods but still force subclasses to fill in the
missing parts.

## 7. What Java concept are you still unsure about?
I am still not fully sure about chained comparators in RankingService. I understand
that .sorted(Comparator.comparing(...).reversed()) sorts by one value from high to
low, and .thenComparing(Player::getUsername) breaks ties by username. But I am not
completely clear on how the JVM decides the final order when two players have exactly
the same score and the same username, or how the method reference Player::getUsername
works internally. I can use it and it gives the right result, but I cannot fully
explain every step yet.

## 8. Did AI make the project easier, harder, or both? Explain.
It made the project easier overall. Many times I knew in my head what effect I wanted,
but I could not turn it into code by myself, and the AI could finish it very quickly.
But I still needed to spend time to learn and understand the code, and to find the
problems in it, so it was not completely free.

## 9. Which parts of the final project were mainly written by you?
The parts mainly written by me are InputHelper, ConsoleUtil, the design.md document, the
changes to plan.md, this reflection, and all of the design decisions.

## 10. Which parts were mainly generated or heavily assisted by AI?
The model classes, the four service classes, the Main menu, the persistence, and the
combat simulation were mostly generated by AI. In fact most of the code was mainly done
by AI, and then I reviewed and tested it again by myself.
